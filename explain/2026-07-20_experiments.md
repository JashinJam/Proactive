# 7月20日实验与工程工作讲解

## 1. 这一天解决的不是一个问题，而是三个相连的问题

7月19日结束时，项目已经形成两个不同定位的决策系统：

- D3是正式科学基线，五折OOF Macro-F1为0.6690；
- D4是面向排行榜的冻结工程候选，参考OOF Macro-F1为0.6846。

但当时还有三件事没有解决：

1. D4还只是项目内部能够运行的实验代码，未验证能否安全地接收组织方隐藏测试输入并生成正式提交文件；
2. U0只有A评测员的分析，尚不清楚语言问题是否只是一个人的评分习惯；
3. U1-V已经发现语言生成严重依赖历史assistant话语，但还没有尝试解决没有历史时的正文冷启动。

因此7月20日依次完成了：D4提交入口审计、U0双人评分聚合、U2早期正文和视觉事实实验。

## 2. D4提交入口与容器前置审计

### 2.1 为什么要做

原有D4运行程序是实验复现工具。它把公共验证集路径和文件哈希写在配置中，还会保存大量实验记录并调用评分器。这适合证明实验可复现，却不能直接处理比赛方以后挂载的隐藏测试文件。

所以这里需要的不是重新训练D4，而是在D4外面增加一层输入输出适配程序。这里的adapter只是“接口转换层”，不是LoRA，也不是增加了一套神经网络。

### 2.2 新入口接收什么

隐藏测试每个session必须提供：

- `video_path`：视频路径；
- `video_intervals`：每个候选chunk的起止时间；
- `query`：用户要完成的任务；
- `domain`：任务领域；
- `dialog`：每个chunk开始前能够看到的累计对话历史。

`dialog`必须和`video_intervals`逐chunk对齐。D4的8个对话阶段特征依赖官方历史，所以隐藏输入如果没有dialog，当前D4就无法按冻结定义运行。

正式入口默认拒绝任何带`answers`的输入，防止隐藏答案误入模型。本地使用公共验证集检查入口时，可以显式打开审计开关，但程序仍会先删除answers，再把数据交给模型。

### 2.3 完整推理过程

入口先检查字段、对话长度和是否含答案，然后根据实际挂载路径生成临时运行配置。之后直接调用已经验证过的D4 shared-vision推理程序：

1. 只读取当前和过去的视频帧；
2. 用InternVL生成原始正文，并提取tag margin和1024维hidden；
3. 构造D1的18个因果数字和D4新增的8个对话阶段数字；
4. 交给冻结的1052参数线性head决定interrupt或silent；
5. 最终只保留官方需要的`video_path`和`answers`两个字段。

输出前还会检查session顺序、chunk数量和每条答案的标签。文件先写到临时位置，全部成功后再一次性替换正式结果，避免程序中途出错留下一个看似完整、实际上只写了一半的提交文件。这就是“原子写入”。

### 2.4 参数和许可核对

- InternVL3.5-1B基座：1,060,897,792参数；
- D4线性head：1,052参数；
- 总参数和实际参与推理的参数均为1,060,898,844，约10.61亿；
- Small组上限为20亿，因此通过；
- 基座模型许可证为Apache-2.0。

当前项目还没有顶层源码许可证。模型许可证可以登记为Apache-2.0，但项目源码最后采用Apache-2.0、MIT或其他许可，需要项目所有者决定，不能由实验自动替用户选择。

### 2.5 做了哪些检查

CPU preflight先检查700个session、9935个chunk的路径、哈希、参数和配置，不加载GPU模型。随后在空闲GPU上跑了1个session、10个chunk的真实入口闭环：

- 总墙钟时间44.206秒；
- 峰值显存约3.47GB；
- 预测6次interrupt；
- 与7月19日冻结D4 smoke的预测文件逐字节相同；
- prompt、原始回复、tag margin、hidden、8个对话特征、logit、决策和最终答案全部10/10一致；
- 最新正式报告记录的R0/D1/D3/D4回归测试为48/48通过。

这说明新包装没有改变模型行为。它不是新的效果实验，10个chunk也不能用来估计排行榜分数。

### 2.6 0.6846、0.7393和最终Docker的关系

0.6846来自五折OOF。每个session由没有用这个session训练过的fold head预测，因此它是目前更可信的公共集效果参考，但整份OOF预测来自5个head。

最终隐藏测试只能装一个模型，所以Docker会使用在全部700个公共session上重新拟合的单一D4 head。这个head在同一批训练数据上的0.7393只说明拟合过程正常，不能当作隐藏测试成绩。

### 2.7 还没有完成什么

- 没有改变D4特征、阈值、帧数或历史长度；
- 没有向排行榜或镜像仓库上传任何东西；
- 还要合入同学最终选定的帧采样和历史配置，并重新做等价检查；
- 按当前项目记录，官方Docker模板发布后还要适配固定路径、启动命令、健康检查和资源限制；
- 还要选择项目顶层源码许可证。

## 3. U0双评测员分析

### 3.1 为什么不能只看A的结果

A的结果已经足够发现fallback和早期正文问题，但人工分数可能混入个人尺度。例如一个人把“画面大致相关”打4分，另一个人只有在每个动作都能直接看见时才打4分。加入B不是简单为了多算一个平均数，而是要知道哪些结论两个人都认可，哪些指标的定义还没有对齐。

U0的200条仍是故意平衡抽取的困难样本，不代表9935个chunk的自然比例。A、B各评200条，共400行评分。程序验证了每条恰好有A、B各一份，没有遗漏、重复或错位。

### 3.2 两个人共同确认了什么

对160条模型实际说话的样本，两人平均后的内容综合分为2.3063/5：

- fallback：1.5413；
- 有具体正文的nonfallback：3.0713。

两人对“fallback是否空泛”判断非常一致，generic标记一致率98.13%，kappa为0.9625。因此“Please continue with the next step”不是合格指导，这个结论非常稳定。

但具体正文也有代价：

- fallback幻觉率约0.63%；
- nonfallback幻觉率约13.13%。

fallback看起来较少幻觉，主要是因为它几乎没有可被验证的具体内容，并不代表它质量高。

### 3.3 哪个位置最需要修

- first位置内容综合分2.7421；
- second位置只有1.7857，是最弱位置；
- 2到4位置为2.3256，但幻觉率11.54%，在这些位置中最高；
- 更后面的位置大约在2.29到2.37之间。

这说明下一步不应泛泛地处理所有正文，而应优先研究second和2到4：second主要是模型无法启动正文，2到4则开始会说具体内容，但更容易说错。

### 3.4 两个人在哪些地方意见一致，哪些地方不一致

correctness和specificity的加权kappa分别为0.6655和0.7050，说明两人对“是否正确、是否具体”的排序有较好一致性。

groundedness则完全不同：

- A平均2.4375；
- B平均4.0125；
- 加权kappa只有0.0508。

kappa可以理解为“扣除碰巧一致以后，两人真正有多一致”。0.05接近没有形成稳定共同尺度。因此，不能把A和B的groundedness简单平均成3.225，然后声称视觉落地已经达到中等水平。

hallucination也存在尺度差异：A标记10%，B标记3.75%。unsafe方面A标记5%，B为0%。这些低频项目即使表面一致率较高，也必须查看原始分歧，不能只看一个平均数。

两人在“是否应该介入”上完全一致的比例为68%，kappa为0.408，46条出现一人选yes、另一人选no。这也再次说明人工使用判断和官方标签不是同一个概念。

宽口径规则把162/200条放入分歧清单，其中很多只是某一项相差2分，不能解读成81%的样本都不可用。这个清单的作用是找评分定义问题，不是重新计算模型错误率。

### 3.5 怎样得到U2的21条样本

为避免人工挑选有利案例，U2使用满足四个条件的完整交集：

1. 位于second或2到4；
2. A和B都认为这里应该介入；
3. D4五折OOF判断为interrupt；
4. 最终单一D4 head也判断为interrupt。

最终得到21个不同session，其中second 9条、2到4有12条，四个领域均有覆盖。这里只使用A/B的yes共识，不按内容分挑高分或低分。

## 4. U2早期正文与视觉事实实验

### 4.1 为什么做U2

U1-V已经发现，删除历史assistant话语以后，模型几乎全部停止生成具体正文。U0双评又确认second最差。因此U2要回答：

1. 没有历史时的fallback是否真是正文冷启动问题；
2. 当前chunk画面是否真正支撑生成；
3. 能否先让模型写一句“当前直接看见什么”，再依据这句事实生成指导，用视觉事实代替历史话语的脚手架。

这里的“事实先行”可以理解成一个非常轻的中间步骤：先观察，再指导。它不是完整planner，也没有输出任务步骤、完成进度和长期计划。

U2固定D4已经作出的interrupt决定，只改变interrupt后面的正文。这样如果输出变化，就能确定变化来自语言生成方法，而不是介入阈值一起改变。因为决策完全不变，所以没有调用官方决策评分器，D3和D4的Macro-F1也都不变。

### 4.2 视觉事实怎样生成

同一个InternVL先单独看任务问题和当前interval的16帧，提示它最多输出三个直接可见的对象、动作或状态：

- 不给建议；
- 不参考历史步骤；
- 不推断动作已经完成；
- 看不清时输出`unclear`；
- greedy生成，最多32个token。

得到的事实再作为文字块送入正文生成。事实和正文复用同一个1B模型实例，不是同时加载两个模型，也没有新增训练参数。正文最多生成64个token。

### 4.3 六个对照视图

| 视图 | 历史assistant话语 | 视频 | 注入预测事实 |
|---|---|---|---|
| full_history | 最近4轮 | 截至当前的累计视频 | 否 |
| no_current_video | 最近4轮 | 只看以前chunk | 否 |
| query_only_full_video | 删除 | 截至当前的累计视频 | 否 |
| query_only_current_video | 删除 | 只看当前chunk | 否 |
| facts_full_history | 最近4轮 | 累计视频 | 是 |
| facts_query_current | 删除 | 当前chunk | 是 |

这六个版本分别检验历史、当前画面和事实文字的作用。每个样本的两个facts版本必须复用同一句预测事实，避免事实本身不同造成混淆。

### 4.4 自动结果

| 视图 | 具体正文 | fallback |
|---|---:|---:|
| full_history | 13/21 | 8/21 |
| no_current_video | 14/21 | 7/21 |
| query_only_full_video | 0/21 | 21/21 |
| query_only_current_video | 3/21 | 18/21 |
| facts_full_history | 13/21 | 8/21 |
| facts_query_current | 6/21 | 15/21 |

结果首先再次确认历史依赖。完整视频仍在，但删掉assistant历史以后，累计视频版本21条全部fallback。只保留当前视频时偶尔能生成3条，但整体仍然失败。

第二，删除当前chunk画面以后，具体正文反而从13条变成14条。这不代表视频有害，只能说明当前模型是否愿意开口，主要不是由当前画面决定。历史可以让模型继续说，即使当前画面被删除。

第三，加入预测事实以后，无历史、仅当前视频的具体正文从3条增加到6条，净救回3条，即提升14.29个百分点。实验开始前规定至少提升20个百分点才进入优先候选，因此没有通过门槛。

在保留完整历史时，加入事实没有救回任何fallback。21条中8条措辞发生变化，但能说话的仍是13条。事实块只能局部修改内容，不能替代历史提供的生成结构。

### 4.5 为什么21条事实都非空，仍不能说事实提取成功

fact pass的21条全部生成了文字，没有输出`unclear`，平均10.95个英文单词。但很多所谓事实实际是命令或任务常识，例如：

- `Place a new doormat grip pad on the floor.`
- `Fold the paper in half, place the string on top, and pull to reveal symmetrical patterns.`
- `Put tuna salad in the sink.`

它们未必是当前画面中可以直接确认的观察。有的已经包含下一步动作，有的可能识别错误，有的来自任务名称而不是视频。

被事实救回的正文也可能扩写出画面没有支持的内容。例如视觉事实只有“把纸对折”，正文却继续写折角、形成翅膀等整套蝴蝶步骤。它虽然不再是fallback，但可能从“没说内容”变成“说了很多未经当前画面支持的内容”。

因此nonempty只表示模型开口了，不表示说对了；文字相似度和关键词重合也不能证明视觉落地。

正式运行完成：

- 21条冻结R0完整视图精确回放；
- 21次事实生成；
- 6个视图乘21条，共126次正文生成；
- D4决策改动为0；
- 总运行约208.88秒；
- 峰值显存约3.48GB。

又独立运行了第二次。自动分析文件逐字节一致，去除自然变化的运行时间后，逐条记录、盲评项目、答案键和评分表都一致。

### 4.7 为什么还要人工评分

U2已经生成两套隐藏版本名称的盲评包：

- 126条正文候选，A/B共252个评分位置；
- 21条预测事实，A/B共42个评分位置。

正文评审新增三项：

- `current_visual_support_1_5`：正文里的对象、动作和状态是否由当前chunk直接支持；
- `unsupported_visual_claim_flag`：是否包含当前画面不支持的具体说法；
- `stale_history_flag`：是否只是在重复旧指导，已经落后于当前进度。

事实包单独评“是否正确”和“是否遗漏当前最重要的任务相关事实”。在正式双评前，要先用少量独立示例统一评分边界，因为U0已经证明两个人对groundedness的理解不同。

## 5. 7月20日之后的路线

1. D3继续是正式科学基线0.6690，D4继续是冻结的榜单工程候选；U2不修改决策分数。
2. D4入口可行性已经验证，等待同学输入策略、项目许可证和官方Docker模板；任何外部提交仍需明确授权。
3. U2 v1没有通过自动门槛，不能在同一21条上继续改提示词、帧数或历史长度，再把更好的结果当作可靠验证。
4. 先校准视觉支持评分规则，再完成U2的事实和正文双人盲评。
5. 只有人工结果证明事实正确、当前视觉支持上升，同时不增加无依据说法和陈旧历史，才值得在新的冻结样本上设计U2 v2。
6. U1-B和独立的no-state/oracle-step/oracle-full状态盲评仍未完成，所以显式planner是否有效仍没有结论。
7. S1大规模状态标注、计划粒度实验和GRPO继续暂停，避免在历史依赖尚未解决时强化错误机制。

## 6. 一句话总结

7月20日先证明D4已经可以安全接收未来隐藏测试输入，再用双评审确认“早期正文冷启动”是真问题，同时发现旧的视觉落地评分尺度并不可靠；随后U2尝试用“先看当前画面并写事实，再生成指导”替代历史话语，确实多救回3条正文，但没有达到预设门槛，而且事实本身可能变成命令或幻觉。因此当前最合理的选择不是继续调提示词，而是先用更明确的视觉支持规则完成人工核查。
