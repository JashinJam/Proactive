# R1 技术讲解：Oracle 紧凑程序状态协议试验

> 更新时间：2026-07-15  
> 对应实验：20260714_internvl35_1b_oracle_state_r1_pilot_v1  
> 实验状态：协议试验完成，R1 科学门槛未通过  
> 说明：本文用于通俗理解。正式事实和数值以实验报告、冻结产物及官方 scorer 为准。

## 1. 一句话理解 R1

R1 没有更换模型，也没有训练模型。它在 R0 的原始输入旁边增加一份由标注者提供的、严格因果的“当前任务进展说明”，然后比较模型是否因此更准确地决定何时打断用户。

它真正想回答的是：

> 如果我们已经能近乎理想地告诉 1B 模型“用户现在做到哪一步、哪些完成条件已经出现、哪里还没完成或做错了、下一步是什么”，模型能否找回 R0 漏掉的 interrupt，同时不制造太多错误打断？

R1 的最终答案是：

> 当前四 session、50 chunk 的零样本文本状态表示没有产生 Macro F1 净收益。完整状态让模型更愿意开口，找回 4 个漏报，同时新增 4 个误报，收益与代价完全抵消。

这不是“程序状态没有价值”的证明，只说明当前样本规模、状态序列化方式和冻结 1B 生成决策接口没有出现正向净信号。

## 2. R1 相比 R0 改了什么

R0 的输入是：

~~~text
截至当前的视觉帧
+ 官方 system prompt
+ query
+ 最近最多 4 条 dialog turn
~~~

R1 的输入是：

~~~text
完全相同的 R0 输入
+ 一个追加在 system prompt 末尾的 <procedural_state> 状态块
~~~

以下内容全部保持不变：

- InternVL3.5-1B-HF checkpoint；
- 1,060,897,792 个模型参数；
- BF16 和 SDPA；
- 每个 interval 最多 16 帧；
- 累计视觉输入最多 32 帧；
- 448 x 448 分辨率；
- 最近 4 条 dialog turn；
- greedy decoding；
- max_new_tokens = 64；
- R0 的输出 canonicalizer；
- 官方 Macro F1 scorer。

因此 R1 是 controlled experiment，即受控实验。主要变量只有追加的状态文本。

## 3. 几个核心术语

### 3.1 Procedural state

procedural state 是“程序性任务状态”。它不是只识别画面里有什么物体，而是描述：

- 总目标是什么；
- 当前应该执行哪一步；
- 当前步骤尚未开始、正在进行、已完成、偏离还是已经恢复；
- 看到了哪些完成证据；
- 看到了哪些未完成或错误证据；
- 下一步是什么。

例如“桌上有花生酱”只是视觉事实；“当前正在准备配料，碗还没有准备好，因此准备步骤未完成”才是程序状态。

### 3.2 Compact state

compact 是“紧凑的”。R1 不把整份长计划和全部视频描述塞给模型，只提供当前决策最相关的少量字段，降低输入长度和冗余。

### 3.3 Oracle state

oracle 在这里不是另一个神奇模型，而是由标注者按照严格信息边界给出的近似理想状态。标注者可以看到当前 interval 结束前的画面，但不能看 gold answer、未来视频、未来 dialog 或 R0 错误。

它用于测一个乐观上限：如果状态信息已经很好，现有决策模型是否会利用它。

这个“上限”不是数学上保证更高的 upper bound。因为模型可能误解文本、忽略证据或被额外 prompt 干扰，所以 oracle state 仍可能让分数下降。

### 3.4 Non-deployable

non-deployable 表示不可直接部署。真实测试时不会有人逐 chunk 手工写状态，因此必须最终训练或实现一个自动 causal state updater。R1 尚未构建这个组件。

### 3.5 Pilot

pilot 是小规模先导试验。它用于验证协议能否运行、是否存在很强的初步信号，不用于估计完整数据集或隐藏测试集上的真实成绩。

## 4. 为什么只选择 4 个 session

逐 chunk 标注程序状态成本很高，所以 R1 先从四个 domain 中各选一个 session：

| 输入索引 | Domain | Task | Chunks |
|---:|---|---|---:|
| 14 | Chef | Making peanut butter oat balls | 14 |
| 123 | Tutorial | How to loosen a tight screw | 10 |
| 326 | Handyman | Adjusting loose recessed light trim | 12 |
| 687 | Arts and Crafts | Creating a simple finger painting | 14 |
| 合计 | 4 个领域 | 4 个任务 | 50 |

选择算法名为 domain_stratified_sha256_first_k：

1. 按 domain 分组；
2. 对 seed、video_path 的组合计算 SHA256；
3. 每个 domain 按 hash 排序；
4. 取排序第一的一个 session；
5. 最后恢复源 JSONL 顺序。

SHA256 在这里是确定性哈希函数，可以把文本映射成固定长度指纹。选择规则不查看答案、R0 预测或错误类型，因此避免了故意挑选“状态最容易帮助”的样本。

但 label-independent 不等于统计上有代表性。选完后才发现这 50 个 chunk 有 30 个 interrupt、20 个 silent，interrupt 占 60%，不同于完整公开集的 53.87%。

## 5. Oracle 标注怎样制作

标注分为静态计划和动态状态两层。

### 5.1 静态计划

静态 plan 只能根据 task 和 query 编写，不允许先看视频、dialog 或 answers。它包含一组步骤，每一步有：

- id：步骤编号；
- text：步骤内容；
- completion_cues：什么现象通常意味着完成；
- incompletion_cues：什么现象通常意味着还没完成。

例如花生酱燕麦球任务中的一个步骤：

~~~text
current step:
Measure and add the oats, peanut butter, sweetener, and mix-ins to the bowl.

completion cue:
All measured ingredients are together in the mixing bowl.

incompletion cue:
One or more ingredients remain outside the bowl or unmeasured.
~~~

cue 是判断线索，它描述“如果看见什么，通常说明某种状态”。它是一般规则，不代表该现象当前真的已经发生。

### 5.2 动态状态

动态 state 按 chunk 时间顺序标注。标注者每次只能使用：

- task 和 query；
- 当前 dialog[i]；
- 时间不晚于当前 interval 结束点的画面；
- 此前已经按顺序观察过的内容。

每个标注包提供当前 interval 内均匀采样的 4 帧，并明确当前可见时间范围。包内不导出 answers，也不导出未来 dialog。

标注完成后，程序才使用正式 R0 的 16/32 帧策略进行模型推理。标注时的 4 帧 contact sheet 与模型的视觉输入不是同一个东西，前者服务于人工状态标注，后者服务于 InternVL 推理。

## 6. 状态字段逐项解释

| 字段 | 中文含义 | 注意事项 |
|---|---|---|
| goal | 总体目标 | 必须等于源 query |
| current_step_id | 当前步骤编号 | 模型最终看到的是步骤文字，不是 id |
| progress | 当前进度类别 | 五个固定值之一 |
| completion_cues | 静态完成线索 | 假设性判断规则，不是已观察事实 |
| incompletion_cues | 静态未完成线索 | 同样是一般规则 |
| completion_evidence | 已看到的完成证据 | 必须来自当前或过去画面 |
| incompletion_or_error_evidence | 已看到的未完成或错误证据 | 可以描述缺失、错误或偏离 |
| next_step_id | 下一步骤编号 | procedure 完成后可以为 null |
| confidence | 标注者对状态解释的信心 | 0 到 1，不是模型预测概率 |
| last_update_chunk | 动态状态最近更新的位置 | 状态不变时允许沿用旧值 |
| observed_through_sec | 标注看到的最晚时间 | 必须精确等于当前 interval 结束时间 |

progress 的五个取值：

| 英文 | 中文解释 |
|---|---|
| not_started | 当前步骤尚未开始 |
| ongoing | 正在执行 |
| complete | 已完成 |
| deviated | 偏离计划、做错或出现明显问题 |
| recovered | 已从此前偏离中恢复 |

observed_through_sec 只用于因果校验，不会写入给模型的 state block。完整 plan 中也只有 current step 和 next step 会暴露给模型，不会一次塞入所有步骤。

## 7. 怎样防止标签泄漏

leakage 是信息泄漏，指模型间接或直接获得本不该看到的答案。

R1 的标注验证器强制检查：

- 4 个 session 和 50 个 chunk 必须完整覆盖；
- video_path、query 指纹必须与源数据一致；
- 每个状态的 observed_through_sec 必须等于当前 interval 结束点；
- current_step_id 和 next_step_id 必须来自静态 plan；
- progress 必须属于固定枚举；
- confidence 必须在 0 到 1；
- last_update_chunk 不能晚于当前 chunk，也不能倒退；
- 动态字段变化时必须声明更新；
- 状态文字不能出现 $interrupt$、$silent$、gold label、should interrupt 等目标提示；
- 推理行会物理删除 answers 字段。

所有 200 次生成结果写入并 fsync 后，程序才生成 evaluation_golden_subset.jsonl 并调用官方 scorer。

这保证状态没有直接抄标签，但不能保证标注完全没有主观误差。oracle 在这里表示高质量因果标注，不表示绝对正确真理。

## 8. 四个状态变体和一个 R0 参照

R1 实际比较五列：

| 变体 | 模型额外看到什么 | 目的 |
|---|---|---|
| r0_frozen | 什么都不加，而且不重新生成 | 保留原始 R0 精确参照 |
| null | 只有 status: unavailable | 测量状态包装器本身的影响 |
| step | 当前步骤文字 | 测当前 step 是否有用 |
| cues | 当前步骤和静态完成/未完成 cues | 测静态判断标准是否有用 |
| full | cues 加目标、进度、实际证据、下一步、信心和更新位置 | 测完整紧凑状态 |

r0_frozen 直接从冻结 R0 产物中抽取同四个 session，不重新运行模型。这避免了无意改变 R0。

null 是 prompt-scaffold control，即“提示词外壳对照”。full 应主要和 null 比，因为二者都有 procedural_state 外壳。r0_frozen 用来判断仅添加外壳是否已经改变行为。

null 并不是与 full 完全等长的 placebo。placebo 是安慰剂对照。null 只控制外壳存在，无法控制额外 token 数量和具体语义，所以 R1 仍存在 prompt length confound，即提示词长度混杂。

## 9. 状态块的精确格式

状态不会成为额外 user 消息，而是追加到原始官方 system prompt 的末尾：

~~~text
[逐字不变的官方 system prompt]

<procedural_state>
...
</procedural_state>
~~~

尖括号标签是一种 XML-like delimiter，即类似 XML 的边界标记，用于告诉模型中间是一块独立结构化信息，但它并不是真正经过 schema 解析的 XML。

### 9.1 Null

~~~text
<procedural_state>
status: unavailable
</procedural_state>
~~~

### 9.2 Step

~~~text
<procedural_state>
current_step: Gather a mixing bowl, measuring tools, oats, peanut butter, a sweetener, and any mix-ins.
</procedural_state>
~~~

### 9.3 Cues

~~~text
<procedural_state>
current_step: Gather a mixing bowl, measuring tools, oats, peanut butter, a sweetener, and any mix-ins.
completion_cues: The bowl, tools, and required ingredients are within reach.
incompletion_cues: A bowl, measuring tool, or required ingredient is missing.
</procedural_state>
~~~

### 9.4 Full

第 14 号 session 的 chunk 0 实际 full block 是：

~~~text
<procedural_state>
current_step: Gather a mixing bowl, measuring tools, oats, peanut butter, a sweetener, and any mix-ins.
completion_cues: The bowl, tools, and required ingredients are within reach.
incompletion_cues: A bowl, measuring tool, or required ingredient is missing.
goal: How do I make peanut butter oat balls?
progress: ongoing
completion_evidence: Open oats, peanut butter, and measuring items are visible.
incompletion_or_error_evidence: A mixing bowl and all remaining ingredients are not yet clearly arranged.
next_step: Measure and add the oats, peanut butter, sweetener, and mix-ins to the bowl.
confidence: 0.88
last_update_chunk: 0
</procedural_state>
~~~

这里没有直接写“应该打断”。模型必须结合原始 system prompt 自己判断这些状态是否值得说话。

## 10. 加入状态后模型输入有多长

使用上述真实 chunk 0、相同 16 帧和相同 processor 还原得到：

| Variant | 总 input token | 视觉 token | 文本和结构 token |
|---|---:|---:|---:|
| r0 | 4,388 | 4,096 | 292 |
| null | 4,404 | 4,096 | 308 |
| step | 4,425 | 4,096 | 329 |
| cues | 4,458 | 4,096 | 362 |
| full | 4,546 | 4,096 | 450 |

图像输入完全相同，变化只发生在 system context 的文字部分。full 比 null 多 142 个 token。这再次说明 null 控制了包装形式，但没有完全控制长度。

## 11. 每个 chunk 的实际推理顺序

对每个选中 session：

~~~text
读取当前 interval 的帧一次
  -> 加入累计帧并压到最多 32 帧
  -> 用同一份 model_frames 跑 null
  -> 用同一份 model_frames 跑 step
  -> 用同一份 model_frames 跑 cues
  -> 用同一份 model_frames 跑 full
  -> 每个输出都使用 R0 canonicalizer
~~~

所以总生成次数是：

~~~text
50 chunks x 4 state variants = 200 次 generate
~~~

四个变体共享完全相同的帧、query、dialog、模型和解码配置。greedy decoding 不使用随机抽样。每次 generate 独立执行，不共享跨变体或跨 chunk 的 KV cache。

r0_frozen 不包含在这 200 次中，因为它来自已经冻结的 R0 文件。

## 12. 一个真实输入输出例子

第 14 号 session 的 chunk 0，full 已明确告诉模型：

- 当前正在准备工具和配料；
- 已看到燕麦、花生酱和量具；
- 尚未清楚看到 mixing bowl 和所有配料就位；
- 下一步是测量并放入所有材料。

模型 raw response：

~~~text
Measure and add the oats, peanut butter, sweetener, and mix-ins to the bowl.
~~~

它在内容上给出了下一步，但没有 $interrupt$ 标签，因此最终：

~~~text
answer: $silent$
normalization: malformed_response_scored_as_silent
~~~

四个 full 首 chunk 全部发生了类似问题。其中拧螺丝样本甚至生成：

~~~text
$Iinterrupt$ I need to apply firm downward pressure...
~~~

因为标签拼成了 $Iinterrupt$，仍被官方按 silent 处理。

## 13. 状态正确也不等于决策正确

第 14 号 session 的最后一个 chunk 描述：

~~~text
progress: deviated
error evidence: The portion remains loose and crumbly instead of holding a ball shape.
next step: Arrange the finished oat balls and chill or serve them.
~~~

模型却输出：

~~~text
$interrupt$Chill the peanut butter oat balls in the fridge...
~~~

这说明模型可能抓住 next_step，却忽略当前“松散、无法成球”的错误证据。理想状态只是提供信息，仍需要 state-aware decision policy 学会如何权衡：

- 出错时应该先纠正，不能机械进入下一步；
- ongoing 不总是值得打断；
- complete 有时应该沉默，有时可能需要确认；
- next_step 不是无条件立即播报的指令。

## 14. 官方子集结果

这 50 个 chunk 有 30 个 gold interrupt 和 20 个 gold silent：

| Variant | Macro F1 | Int. Precision | Int. Recall | Silent Recall | TP/FP/TN/FN | Pred. interrupt |
|---|---:|---:|---:|---:|---|---:|
| r0_frozen | 0.5398 | 0.6842 | 0.4333 | 0.7000 | 13/6/14/17 | 38% |
| null | 0.5169 | 0.6875 | 0.3667 | 0.7500 | 11/5/15/19 | 32% |
| step | 0.4949 | 0.6667 | 0.3333 | 0.7500 | 10/5/15/20 | 30% |
| cues | 0.3592 | 0.4545 | 0.1667 | 0.7000 | 5/6/14/25 | 22% |
| full | 0.5169 | 0.6250 | 0.5000 | 0.5500 | 15/9/11/15 | 48% |

这些是四 session 子集诊断，不能和 R0 在完整 700 session 上的 0.4630 直接比较。r0_frozen 的 0.5398 只说明这四个样本对 R0 相对更容易或类别构成不同。

## 15. 为什么 full 和 null 分数完全相同

从 null 到 full：

~~~text
TP：11 -> 15，增加 4 个正确打断
FP： 5 ->  9，增加 4 个错误打断
TN：15 -> 11，损失 4 个正确沉默
FN：19 -> 15，找回 4 个漏报
~~~

full 的 interrupt recall 从 0.3667 提高到 0.5000，说明状态确实让模型更愿意说话；但 silent recall 从 0.7500 降到 0.5500，说明它也更频繁地无必要打断。

null 的两类 F1：

~~~text
interrupt F1 = 0.4783
silent F1    = 0.5556
Macro F1     = 0.5169
~~~

full 的两类 F1 几乎正好交换：

~~~text
interrupt F1 = 0.5556
silent F1    = 0.4783
Macro F1     = 0.5169
~~~

因此 R1 观察到的是 class tradeoff，即类别之间的取舍，不是净提升。

## 16. Full 具体改变了哪些决定

相对 null，full 改变了 50 个决定中的 14 个：

| 变化 | Gold | Progress | 数量 |
|---|---|---|---:|
| silent -> interrupt | interrupt | ongoing | 5 |
| silent -> interrupt | silent | ongoing | 3 |
| interrupt -> silent | interrupt | not_started | 2 |
| silent -> interrupt | interrupt | deviated | 1 |
| silent -> interrupt | interrupt | complete | 1 |
| silent -> interrupt | silent | complete | 1 |
| interrupt -> silent | interrupt | ongoing | 1 |

大多数变化发生在 ongoing，而不是只发生在 complete 或 deviated。这说明当前 full state 没有形成简单可靠的“完成则沉默、错误则打断”规律，而是整体改变了模型的说话倾向。

## 17. 为什么 cues 反而伤害最大

cues 只提供当前步骤以及“完成时通常看到什么、未完成时通常看到什么”，却不告诉模型当前实际观察到了哪一种。

例如：

~~~text
completion_cues: The mixture holds together.
incompletion_cues: Dry oats remain separate.
~~~

这两句话都是条件性描述，不是当前事实。1B 零样本模型可能：

- 把假设性 cue 当成已经发生的画面描述；
- 因为同时看到完成和未完成条件而无法判断；
- 把长状态块当成操作说明，而不是决策依据；
- 更倾向输出普通指导文本或沉默。

cues 的 predicted interrupt rate 从 null 的 32% 降到 22%，Macro F1 降至 0.3592。上述解释是根据行为作出的合理推断，不是对模型内部机制的直接证明。

## 18. 格式混杂有多严重

四个选中 session 的首 chunk 全部是 gold interrupt：

| Variant | 首 chunk TP/FN | Interrupt recall | 格式错误数 |
|---|---:|---:|---:|
| r0_frozen | 1/3 | 0.25 | 3 |
| null | 2/2 | 0.50 | 2 |
| step | 1/3 | 0.25 | 3 |
| cues | 0/4 | 0.00 | 4 |
| full | 0/4 | 0.00 | 4 |

因此 R1 同时混有两个问题：

1. 状态是否改善了 interrupt/silent 决策；
2. 状态块是否使模型更容易忘记输出合法标签。

正式结果必须保留这种失败，因为官方 scorer 就会如此判定，不能事后把自然语言指导偷偷改成 interrupt。

## 19. Posthoc 诊断是什么意思

posthoc 是“看到正式结果之后再做的事后分析”，只能帮助理解，不能替代预先冻结的正式实验。

如果事后把所有非空非法自然语言都理解为 interrupt：

- repaired r0_frozen Macro F1：0.5994；
- repaired full Macro F1：0.5895。

即使消除格式问题，full 仍没有超过无状态 R0。因此格式错误不是 R1 未获净收益的唯一原因。

## 20. 后续 grammar 约束为何停止

R1 之后尝试过两种 label-independent grammar，即不看标签的输出语法约束：

1. FSM 前缀约束：强制第一个合法序列只能是 $silent$ 或 $interrupt$；
2. width-2 beam：分别计算完整 silent 标签和 interrupt 标签的序列分数。

FSM 是 finite-state machine，中文是有限状态机。beam search 是束搜索，同时保留若干候选序列。likelihood 是模型给序列的相对可能性。

它们在一个 14-chunk Chef smoke test 上几乎全部预测 interrupt：

- 相同 R0 和 null：14 个中 13 个 interrupt；
- step、cues、full：14 个中 14 个 interrupt；
- silent F1 坍缩为 0。

因此“强制标签合法”虽然消除了 malformed response，却破坏了类别平衡。两个 grammar 都在 smoke 阶段停止，没有扩大到 50 chunk，更没有扩大到 700 session。

这说明 token likelihood 不是已经校准好的 decision gate。decision gate 是专门决定是否打断的门控机制。

## 21. R1 真正证明了什么

R1 已经证明：

- label-independent 的四领域样本选择可以复现；
- 无 gold、无未来信息的标注包可以生成；
- 静态 plan 和逐 chunk 动态状态 schema 可以校验；
- null、step、cues、full 可以共享完全相同的因果视频输入；
- 原始 system prompt 可以保持为逐字不变前缀；
- 200 次多变体推理、断点续跑、产物保存和官方评分管线正常；
- full state 会实质改变模型行为，并提高 interrupt recall；
- 当前改变主要是说话率上升，而不是选择性更准确。

R1 没有证明：

- 程序状态总体无效；
- full state 在 700 session 上一定无收益；
- 手工 oracle 状态能自动部署；
- PWR 方法无效；
- 换成训练过的 decision head 后状态仍无效；
- hidden test 上会得到相同结果。

## 22. 为什么暂不扩展到 16 个 session

原计划只有出现清楚、可重复的净收益，才继续花成本扩大 oracle 标注。当前没有达到门槛：

1. 没有状态变体超过 r0_frozen；
2. full 没有超过 null；
3. interrupt recall 增益被 FP 增加完全抵消；
4. cues 在四个 session 上都造成伤害；
5. full 和 cues 的首 chunk 有严重格式混杂；
6. 四个 session 太少，无法判断结果是否稳定。

继续照搬当前序列化方式标 16 个 session，很可能只是更昂贵地重复同一混杂。因此项目先转向更稳定的 decision interface，也就是后来的 D1 决策头；以后应在训练过的门控接口上重新检验状态价值。

## 23. 最准确的结论表述

错误说法：

> R1 证明 plan/state 没用。

正确说法：

> 在四个 label-independent session、50 个 chunk 上，把人工因果 oracle state 作为额外 system 文本交给冻结 InternVL3.5-1B，并继续使用自由生成标签的零样本接口，没有提高 Macro F1。full state 提高 interrupt recall，但同等增加 false positive；样本太小，不能排除更好的序列化、训练过的决策接口或更大复现实验中存在真实状态收益。

## 24. 术语速查

| 英文术语 | 中文解释 |
|---|---|
| procedural state | 当前任务步骤、进展和证据组成的程序状态 |
| compact state | 只保留决策所需少量字段的紧凑状态 |
| oracle | 在严格边界内由标注者提供的近似理想信息 |
| upper-bound pilot | 用理想输入测试潜在上限的小规模试验 |
| deployable | 可以在真实测试时自动运行和提交 |
| state updater | 根据新视频不断更新内部任务状态的模块 |
| cue | 判断完成或未完成的静态线索 |
| evidence | 当前已经观察到的具体证据 |
| progress | 当前步骤的进度类别 |
| variant | 实验中的一个受控版本 |
| null control | 只有外壳、没有有效状态内容的对照 |
| scaffold / wrapper | 包住状态内容的提示词结构 |
| confound | 多个因素同时变化导致无法明确归因的混杂 |
| label leakage | 状态或输入泄露标准答案 |
| non-deployable | 无法在真实测试时自动获得 |
| pilot | 小规模先导实验 |
| posthoc | 看到结果后进行的事后分析 |
| FSM | 用有限状态和转移规则约束输出的有限状态机 |
| beam search | 同时保留多个候选序列的搜索方法 |
| likelihood | 模型给一个 token 或序列的相对可能性 |
| calibration | 模型分数、类别比例和真实发生率是否匹配 |
| decision gate | 专门决定说话还是沉默的门控模块 |
| class tradeoff | 提升一类表现却损害另一类表现的取舍 |
| false positive / FP | 本应沉默却错误打断 |
| false negative / FN | 本应打断却错误沉默 |
| scientific gate | 是否值得进入下一阶段实验的预设门槛 |
