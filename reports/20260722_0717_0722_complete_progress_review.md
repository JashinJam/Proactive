## 1. 先给出全局结论

7 月 17 日以后，我们没有沿着一条路线盲目加模型，而是把原来混在一起的三个问题拆开：

1. **决策问题**：当前时刻应该说话，还是保持安静？
2. **正文问题**：决定说话以后，具体应该说什么？
3. **状态问题**：用户现在做到哪一步、是否完成、是否出错？

这三条线最终给出了一个相互印证的结论：

- 决策分数确实可以继续提高，但当前最可靠的增益主要来自官方 `dialog` 暴露的对话阶段和上一轮发言痕迹，而不是已经证明的视觉步骤理解；
- 模型生成具体正文时也高度依赖历史 assistant 话语，当前画面更多是在局部改变措辞，尚不能稳定决定“能不能说出内容”；
- 冻结 1B 模型不能稳定识别细粒度步骤和进度，当前没有足够证据支持在最后一周继续做学习型 planner；
- D5 虽然把 OOF Macro F1 从 D4 的 `0.6846` 提到 `0.6912`，但收益对数据划分敏感；D6 的低维门槛方案又明确失败，所以不能继续围绕同一公共数据反复搜索；
- D4 已经完成单一决策头、在线状态、GPU 推理和提交入口的工程闭环，是目前最适合进入最终交付的方案；
- 当前真正未闭环的，不再是“还能不能多挤出一点公共集分数”，而是隐藏输入是否真的提供相同 `dialog`、没有 `dialog` 时如何退化、官方 Docker、项目许可证以及实际提交。

用一句话概括这几天的路线变化：

> 我们先证明“跨时间信息有用”，随后发现“有用的信息主要来自官方对话历史”，再尝试继续利用短期动作历史，但稳定性不够；与此同时，视觉状态和 planner 没有通过证据门槛，因此最终从研究扩张转向 D4 工程收口。

这些结果留下三个直接问题：

### 2.1 D1 只看“当前”，没有显式看“变化”

D1 知道当前 chunk 的时间、模型倾向和 hidden，但不知道当前表示相对上一 chunk 改变了多少。

例如，单看一张画面，“手里拿着胶水”可能既表示正在涂胶，也可能表示已经完成涂胶。若能比较前后画面和前后表示，就更容易判断步骤是否刚刚发生变化。这推动了 D3。

### 2.2 决定说话以后，正文经常只是固定句

当 D1 判定 interrupt，但原模型没有生成合格正文时，系统会输出：

```text
Please continue with the next step.
```

这句话格式合法，却几乎不能指导用户。因此需要 U1 把决策固定，只研究正文生成，避免把“什么时候说”和“说得好不好”混在一起。

### 2.3 Planner 是否值得做，不能只靠直觉

PWR 论文强调步骤计划、当前进度和错误恢复，但公开 C1 数据没有直接提供 planner 训练目标。若 1B 模型连“现在是第几步、进行中还是已完成”都无法稳定识别，那么直接训练完整 planner 或上强化学习，只会增加复杂度。S0 因此先做状态识别可行性测试。

## 4. 7 月 17 日：D3 因果动态决策

### 4.1 动机与目标

D1 的 OOF Macro F1 已经达到 `0.6341`，而 D2 两种增加容量的方法都几乎没有收益。这说明问题未必是分类器太小，更可能是输入中缺少“发生了什么变化”。

D3 的目标非常单一：

> 不重新训练 1B backbone，只利用已经冻结的 D1 缓存，加入当前 chunk 相对过去 chunk 的变化，检查能否稳定改善 interrupt/silent。

### 4.2 完成的工作

D3 在 D1 的 1,043 维当前特征上增加：

- 8 个低维变化指标，例如 tag margin 相对上一时刻的变化、hidden 与上一时刻的相似度；
- 1,024 维 `current_hidden - previous_hidden`；
- 最终得到 2,075 维输入和 2,076 参数的线性头。

所有变化都严格按 session、按 chunk 顺序计算。首 chunk 没有过去状态，所以动态特征全部为 0；切换 session 时历史完全清空。

为了避免看到结果后挑方案，实验前固定了四个版本：D1 精确重放、只加动态标量、只加 hidden delta、两者都加。只有最后一个主版本允许晋级。

### 4.3 详细结果

| 版本 | Macro F1 | Interrupt F1 | Silent F1 | 相对 D1 |
|---|---:|---:|---:|---:|
| D1 exact replay | `0.6341` | `0.6352` | `0.6330` | - |
| 只加动态标量 | `0.6551` | `0.6633` | `0.6469` | `+0.0210` |
| 只加 hidden delta | `0.6594` | `0.6734` | `0.6454` | `+0.0253` |
| **两者融合 D3** | **`0.6690`** | **`0.6845`** | **`0.6535`** | **`+0.0349`** |

D3 的混淆矩阵为：

```text
TP / FP / TN / FN = 3560 / 1489 / 3094 / 1792
预测 interrupt 比例 = 50.82%
```

它相对 D1：

- 修复 755 个 FN，也就是原来漏掉、现在成功介入；
- 修复 493 个 FP，也就是原来多说、现在保持安静；
- 同时新引入 360 个 FN 和 534 个 FP；
- 净修复 354 个 chunk。

稳定性方面：

- 5/5 folds 都提升，增益在 `+0.0227` 到 `+0.0506`；
- 4/4 domains 都提升，不过 Chef 只有 `+0.0066`，明显弱于其他领域；
- 首 chunk 只提升 `+0.00036`，non-first Macro 从 `0.6045` 到 `0.6446`；
- 5,000 次 session bootstrap 的 95% 区间为 `[+0.02654,+0.04372]`，完全大于 0。

因此 D3 正式晋级为新的科学 OOF 基线。

### 4.4 发现的问题

事后解释发现，D3 的动态变化不仅包含视频变化，也包含对话历史变化。在 9,235 个非首 chunk 中：

```text
当前 dialog 相对上一 prefix 是否新增 assistant turn
    ==
上一 chunk 的 gold interrupt/silent
```

一致率是 `9,235/9,235`。

一个具体例子：如果上一 chunk 官方答案是 interrupt，下一 chunk 的 `dialog` 中就会多一条 assistant 话语；若上一 chunk 是 silent，dialog 往往不变。模型的 hidden 和 tag margin 因此会发生很明显的不同。D3 可能学到的是“上一轮刚说过话，现在更可能安静”，而不完全是“用户刚完成了某个视觉步骤”。

这不构成未来信息泄漏，因为上一轮历史在时间上确实可见；但它改变了科学解释：

- 可以说 D3 合法利用了历史状态；
- 不能说 `+0.0349` 全部来自视觉程序理解；
- 如果真实部署改成模型自己回写历史，错误可能逐步改变后续输入。

### 4.5 工程闭环与下一步

D3 完成了：

- 全量 refit 和 2,076 参数头序列化；
- 单一全量 head 的 train-fit Macro 为 `0.7544`，只用于确认拟合闭环，不能当成泛化成绩；
- 9,935 个 chunk 在线状态回放，决策 `9,935/9,935` 一致；
- 10-chunk shared-vision GPU smoke，各项中间量和输出 `10/10` 一致；
- 峰值显存约 `3.466 GB`。

这个疑问直接推动了 7 月 19 日的 D3-D：把 dialog 规律显式写成特征，看看不用隐式动态表示能达到多少。

## 5. 7 月 17 日：U1 正式盲标与 Oracle 正文生成

### 5.1 动机与目标

官方 Macro F1 只看标签前缀，不评价 interrupt 后面的句子是否正确。因此 D1/D3 分数提高，并不代表助手真正好用。

U1 固定 D1 的所有 interrupt/silent 决策，只比较四种正文接口：

```text
当前固定 fallback
强制生成正文，但不提供 state
强制生成正文，提供 oracle current/next step
强制生成正文，提供完整 oracle progress/evidence/recovery
```

这样，正文变好或变坏不会和 gate 改变混在一起。

### 5.2 为什么要重做盲态标注

早期 4-session oracle smoke 的标注者已经见过模型输出，存在无意识迎合输出的风险，所以它被降级为工程诊断。

正式版本重新标注 20 sessions、80 个时刻：

- 标注者只看 query、task、当前以前的 dialog 和允许的视频区间；
- 不看模型输出、D1 决策、gold answer 和未来视频；
- 80 个状态中 ongoing/complete/deviated/recovered 为 `47/16/7/10`；
- 平均标注置信度 `0.924`，低于 `0.8` 的 7 条没有事后修改。

### 5.3 完成的工作

每个样本使用完全相同的因果视频、最近 4 个历史 turns、greedy decoding 和 64-token 上限。模型先预填真实的 `$interrupt$` 前缀，再生成正文。

实验验证：

- R0 原始生成 `80/80` 精确重放；
- 9,935 个 D1 决策完全不变；
- 官方 Macro F1 三个版本都保持 `0.6341`；
- 生成耗时约 662 秒，峰值显存约 3.50 GB。

### 5.4 自动结果

| 版本 | 80 条中有具体正文 | fallback/EOS | 平均正文词数 |
|---|---:|---:|---:|
| no-state | 56 | 24 | 16.38 |
| oracle step | 56 | 24 | 16.41 |
| oracle full | 57 | 23 | 15.26 |

Oracle state 会明显改写文本：

- step 相对 no-state 改变 43/80 条；
- full 相对 no-state 改变 53/80 条；
- 但 step 没有多救回任何正文，full 只多救回 1 条。

这说明模型“看到了 state block”，但不能说明改写后的内容更正确。文本变化可能是改善，也可能是根据 oracle 信息说出新的错误。

### 5.5 困境与下一步

自动指标无法评价“指导是否正确、是否具体、是否由画面支持”。因此建立了双人独立盲评协议：两名评测员分别打 correctness、specificity、actionability、groundedness、plan consistency、conciseness、safety，并记录 generic、hallucination、premature 和 unsafe。

state 盲评包共有 240 个候选、480 个 reviewer rows。在双人结论完成前，不能宣称 oracle state 有效，也不能据此启动 predicted state、granularity 或 GRPO。

## 6. 7 月 17 日：S0 结构化状态识别与 S1 准备

### 6.1 动机与目标

U1 回答“给定正确 state，正文会不会变好”；S0 回答另一个更基础的问题：

> 不给动态 state 答案，只给一个四步静态计划，冻结 1B 能否从视频和历史中判断当前步骤、进度和错误？

若这个问题都做不好，就没有理由马上把一个 predicted planner 接到提交系统中。

### 6.2 实验设计与工程问题

80 条输入和 target 被物理分开。模型预测时看不到 target，只在两个视图完成并冻结后才评分：

- `official_dialog`：保留官方 assistant 历史；
- `no_assistant_history`：删除 assistant 历史，视频和其他输入不变。

需要预测：

- step：s1 到 s4；
- progress：未开始、进行中、完成、偏离、恢复；
- error：有或无。

第一次 smoke 暴露了一个工程陷阱：模型天然偏爱数字 token `1`，其次是 `2`、`3`，即使没有任何内容也会有选项先验。若不处理，预测 s1 很多可能只是 token 偏好。

因此 V2 在正式预测前固定“无内容校准”：先只给 query 和静态计划、不给视频和 dialog，计算各候选的天然偏好，再从正式分数中减掉。之后没有再搜索温度、选项映射或提示词。

### 6.3 详细结果

| 视图 | Step Macro | Progress Macro | Error Macro | Mean task Macro | Step-progress joint |
|---|---:|---:|---:|---:|---:|
| official dialog | `0.2226` | `0.1348` | `0.5098` | **`0.2891`** | `0.050` |
| no assistant | `0.2601` | `0.0167` | `0.2024` | **`0.1597`** | `0.000` |

预先规定的最低可用门槛是 Mean Macro `0.35`，两个版本都失败；80 条里没有一条同时把 step、progress、error 三项全部预测正确。

更重要的是预测发生了极端塌缩：

```text
有官方 dialog：64/80 预测为 s4，70/80 预测 complete，71/80 预测 error-present
无 assistant：68/80 预测为 s1，69/80 预测 not-started，74/80 预测 error-absent
```

这意味着模型把“历史里已经有很多 assistant 话语”粗略理解为“任务已经很晚、可能完成或出错”；删除历史以后，又把任务理解为刚开始。它识别的是对话阶段两端，不是细粒度视频状态。

虽然 official dialog 的综合正确性比 no-assistant 高 `+0.1750`，bootstrap 区间 `[+0.1125,+0.2375]`，但这只能说明历史有强信号，不能说明状态解码已经可用。

### 6.4 S1 准备与暂停

S0 之后准备了 S1：32 个新 sessions、444 个连续状态，24 sessions/318 states 训练，8 sessions/126 states 一次性 held-out。计划比较 temporal-only、current D1 和 D3 dynamics 三种小型线性 state decoder。

但截至暂停时只完成 2/32 sessions、23/444 states。随后 Reviewer A、U1-V 和 D3-D 都没有把主要残差定位到真实 step/progress，因此继续标注剩余 421 条的收益不足，S1 被保留资产但暂停执行。

### 7.3 最关键的位置和历史分析

| 位置 | Forced composite | Forced fallback rate | 相对固定句 |

按最近可见 assistant turns 分组：

```text

2 turns：fallback 0%
3 turns：fallback 8.33%
4 turns：fallback 3.23%
```

这揭示了一个比“是否有 oracle state”更直接的问题：模型在只有一条 assistant 历史时几乎说不出来；历史积累到两条以上后，正文突然变得容易生成。

例如，模型可能把上一轮“接下来调整背带”的指导当作程序骨架。到了下一时刻，即使画面证据不强，它也能沿着这句话继续生成；但也可能错误宣布背带已经穿好。

### 7.4 对路线的影响

因此没有立刻继续 S1 标注，而是提出两个更直接的控制：

1. U1-V：删除 assistant history、删除当前视频、遮蔽像素，直接判断模型依赖什么；
2. D3-D：不用 hidden dynamics，只用官方 dialog 的显式统计量，看 D3 增益能重建多少。

## 8. 7 月 19 日：U1-V 与 D3-D 揭示历史依赖

### 8.1 U1-V 的动机和设计

U1-V 对同一批 80 个样本比较：

- 完整历史和视频；
- 删除 assistant 历史；
- 删除当前 interval 视频，只保留过去视频；
- 保持帧数但把所有像素改成中灰。

其他 prompt、decoding、帧数和模型全部固定。

### 8.2 U1-V 结果

| 视图 | Fallback | 相对 full 的文本完全一致 | 平均文本相似度 |
|---|---:|---:|---:|
| full | `30.00%` | `100%` | `1.0000` |
| no assistant history | **`100.00%`** | `30.00%` | `0.3000` |
| no current interval video | `26.25%` | `46.25%` | `0.7734` |
| masked video | `30.00%` | `31.25%` | `0.6479` |

删除 assistant 历史后，原来 56 条能生成正文的样本全部退化为 fallback。这说明历史不是小幅辅助，而是当前正文生成的必要脚手架。

删除当前视频后 fallback 反而少了 3 条，说明当前画面不是决定模型能否继续生成的主要因素。遮蔽视频会明显改写部分句子，表明像素确实参与了生成，但这种改变是否正确没有得到证明。

一个直观例子：完整画面下模型可能说“把烤芝士放入锅中”，删除当前画面后退回“先把一片面包放到锅里”。视觉改变了步骤措辞，但历史仍能独立支撑一条看似合理、可能已经过时的指导。

### 8.3 D3-D 的动机

D3 的 OOF 增益是 `+0.0349`。D3-D 把 dialog 中容易理解的阶段信息直接写成 8 个数字，例如：

- 是否刚刚新增 assistant turn；
- 累计说过多少次；
- 已经连续沉默多少个 chunk；
- 最后一条 assistant 话语长度；
- 最后一条是否带 `$interrupt$`。

这些特征在构造前删除 `answers`，只读取当前时刻已经可见的 dialog。

### 8.4 D3-D 详细结果

| 版本 | Macro F1 | 相对 D1 | 对 D3 增益的重建比例 |
|---|---:|---:|---:|
| D1 replay | `0.6341` | - | - |
| 只看是否新增 assistant | `0.6475` | `+0.0134` | `38.4%` |
| 只用 8 个 dialog-stage | `0.6618` | `+0.0277` | `79.4%` |
| D1 + 是否新增 assistant | `0.6749` | `+0.0408` | `116.9%` |
| **D1 + 8 个 dialog-stage** | **`0.6846`** | **`+0.0505`** | **`144.7%`** |

最高版本的 interrupt/silent F1 为 `0.6893/0.6799`，5/5 folds、4/4 domains 和所有非首位置分组都提高。Bootstrap 相对 D1 的区间是 `[+0.0418,+0.0591]`。

最强特征是“距离上次 assistant 新增已经过去多少个 chunk”。这可以理解为模型学到了官方发言节奏：刚说完通常先安静一段；连续安静较久以后，再次介入的可能性上升。

### 8.5 结论和承接

U1-V 和 D3-D 从正文与决策两个方向给出了同一个答案：当前模型首先依赖 dialog，视觉只做不稳定的局部修正。

因此：

- D3 保留为正式科学基线，因为它按预注册通过全部门槛；
- D3-D 的最高版本原协议规定为机制诊断，不能事后改称科学晋级；
- 但排行榜仍需要利用这个合法、可见且稳定的信号，所以将唯一固定的 8 特征版本封装成 D4 工程候选；
- S1、granularity 和 GRPO 继续暂停。

## 9. 7 月 19 日：D4 从诊断结果变成可运行候选

### 9.1 动机与目标

D3-D 找到了更高的 `0.6846`，但它只是一份五折诊断结果。隐藏测试不能运行五个 fold head，也不能直接读取离线缓存。因此 D4 的目标不是继续调特征，而是把这唯一方案变成一个单一、可在线运行的模型。

### 9.2 固定系统

D4 使用：

```text
D1 的 18 个因果标量
+ 1 个 tag margin
+ 1,024 维当前 hidden
+ 8 个 dialog-stage 标量
= 1,051 个特征
```

线性头只有 1,052 个参数。连同 InternVL3.5-1B，总参数是 `1,060,898,844`，约 `1.060899B`，低于 Small 组 2B 上限。

### 9.3 Full-development refit

五个 OOF fold 都选择 `L2=0.01`，最终 threshold 使用五个 calibration threshold 的中位数 `0.1263874797442615`，没有根据全量训练结果重新挑阈值。

单一 head 在全部 700 public sessions 上拟合后的训练集检查结果为：

```text
Macro F1 = 0.7393
Interrupt/Silent F1 = 0.7336 / 0.7450
预测 interrupt 比例 = 43.96%
```

必须特别说明：`0.7393` 是在训练过的同一批数据上计算，只证明拟合流程正常；它不能替代 OOF `0.6846`，更不能当成隐藏测试预期。

### 9.4 在线和 GPU 验证

D4 在线状态逐 chunk 维护 assistant count 和距上次发言间隔。对 9,935 个缓存 chunk 回放：

- 8 个 dialog 特征最大误差 `0.0`；
- 决策 `9,935/9,935` 一致；
- 最大 logit 差 `2.55e-7`。

随后做 10-chunk GPU smoke：raw response、prompt、tag margin、hidden、dialog features、decision 和最终 answer 全部 `10/10` 精确一致；峰值显存约 `3.466 GB`。

D4 因而成为当前最强的排行榜工程候选，但科学口径仍然是：D3 是正式基线，D4 是基于公共诊断的工程选择。

## 12. 7 月 20 日：U2 视觉事实先行诊断

### 12.1 动机

U1-V 证明没有 assistant history 时模型几乎说不出来。U2 尝试一个很轻的替代：

> 先让同一个 1B 模型只看当前画面，写出当前可见事实；再把事实文字放入正文生成，看看能否代替历史提供语言脚手架。

这不是完整 planner，只是“先观察，再指导”。D4 gate 完全固定，所以 U2 不改变任何官方 Macro F1。

### 12.2 六个对照

| 视图 | Assistant history | 视频 | 预测事实 |
|---|---|---|---|
| full history | 有 | 累计因果视频 | 无 |
| no current video | 有 | 只到上一 interval | 无 |
| query-only full video | 无 | 累计因果视频 | 无 |
| query-only current video | 无 | 仅当前 interval | 无 |
| facts full history | 有 | 累计因果视频 | 有 |
| facts query current | 无 | 仅当前 interval | 有 |

### 12.3 自动结果

| 视图 | 具体正文 | Fallback |
|---|---:|---:|
| full history | `13/21` | `8/21` |
| no current video | `14/21` | `7/21` |
| query-only full video | `0/21` | `21/21` |
| query-only current video | `3/21` | `18/21` |
| facts full history | `13/21` | `8/21` |
| facts query current | `6/21` | `15/21` |

事实块在“无历史、仅当前视频”条件下把具体正文从 3 条提高到 6 条，净救回 3 条，即 `+14.29pp`，低于预先规定的 `+20pp` 门槛。

保留历史时，事实块没有多救回任何一条，只改变了 8/21 条措辞。它能调制内容，但不能替代历史。

### 12.4 为什么“21/21 都生成事实”不代表成功

预测事实 21/21 非空，平均 10.95 个英文词，但人工抽查发现不少内容其实是命令或任务常识，例如：

```text
Place a new doormat grip pad on the floor.
Fold the paper in half, place the string on top...
Put tuna salad in the sink.
```

这些文本未必是当前画面可以直接确认的事实。有时模型从一个简单观察扩写出整套后续步骤，从“没有内容”变成“有很多未经画面支持的内容”。因此 nonempty 不是 correctness，词面重合也不是 grounding。

### 12.5 结论与承接

U2 自动门槛失败，不修改 D4，不在同一 21 条上继续搜索提示词、帧数或历史窗口。虽然盲评包已经准备好，但在当前一周交付优先级下，新的 U2 人评和正文训练继续暂停。

## 13. 7 月 21 日：D5 决策融合与短期动作历史

### 13.1 动机

D3 和 D4 虽然都利用历史，但逐 chunk 决策一致率只有 `82.49%`：

- 两者都正确：5,858 条；
- 两者都错误：2,337 条；
- 只有 D4 正确：944 条；
- 只有 D3 正确：796 条。

这表明两者可能互补。D4 主要使用当前 dialog 阶段；D3 还包含 hidden/tag margin 的变化。D5 因而检验：

> 把 D3 完整动态、D4 dialog-stage 和最近几次历史 action 合在一起，是否能得到稳定增益？

短期 action-history 包括最近 2/4/8 次的介入比例、连续沉默长度、最近两次是 II/IS/SI/SS 等。

例如：

- `II` 表示前两次都 interrupt；
- `IS` 表示先 interrupt 后 silent；
- `SS` 表示连续两次 silent。

这些历史都来自当前可见 prefix，不读取当前或未来答案。

### 13.2 五个版本和结果

| 版本 | Macro F1 | 相对 D4 | Interrupt F1 | Silent F1 |
|---|---:|---:|---:|---:|
| D4 replay | `0.6846` | - | `0.6893` | `0.6799` |
| + dynamic scalar | `0.6847` | `+0.0001` | `0.6914` | `0.6780` |
| + full dynamics | `0.6853` | `+0.0007` | `0.6954` | `0.6753` |
| + action history | `0.6889` | `+0.0043` | `0.6955` | `0.6824` |
| **+ full dynamics + action history** | **`0.6912`** | **`+0.0066`** | **`0.7007`** | **`0.6818`** |

表面上主版本达到本阶段最高分，而且 5/5 folds 为正、3/4 domains 为正。

### 13.3 详细错误变化

D5 相对 D4 改变 1,456 个决策：

```text
修复 FN 402
修复 FP 360
新增 FN 290
新增 FP 404
```

共修复 762 个错误，同时引入 694 个错误，净增加 68 个正确 chunk。244 个 session 变好，218 个变差，238 个不变。正负 session 数很接近。

位置上：

- second：`+0.01415`；
- 2-4：`+0.01334`；
- 5-9：`+0.01824`；
- 10+：反而 `-0.00768`。

领域上 Chef 为 `-0.00205`，其他三个领域为正。

### 13.4 为什么不晋级

主要有三层稳定性问题：

1. 5,000 次 session bootstrap 区间为 `[-0.00025,+0.01356]`，下界略低于 0；
2. 三组新数据划分的 D5-D4 分别为 `+0.0072/-0.0007/+0.0016`，其中 B 为负；
3. 自动审计发现三对完全重复列，部分提升可能来自重复特征改变了正则强度，而不完全是新信息。

例如，数据中一次最多只新增一条 assistant turn，所以“是否新增”和“新增数量”两列完全相同；“距离上次发言的连续沉默长度”也与另一列重复。在线性模型中，重复同一信号相当于改变它受到的正则约束，不能把全部增益都解释成新的动作规律。

实验前规定 bootstrap 下界必须大于 0、三组 stability split 都必须为正。即使只差 `0.00025`，也不能看完结果后放宽门槛。

因此 D5 被保留为“有方向性但对划分敏感”的诊断，不做全量 refit、GPU smoke 或提交集成。它留下的问题是：也许不用把这些历史特征加入 2,100 维头，只需按历史阶段微调 D4 的门槛。这推动了 D6。

D5 全程复用冻结缓存、只在 CPU 上运行，五个版本和三组额外 split 总耗时约 881 秒；连同既有回归共 `54/54` 项测试通过。第二层意义是：这次不晋级来自统计稳定性，而不是代码没有跑通。

## 14. 7 月 21 日：D6 低维结构化门槛校准

### 14.1 动机

D5 可能因为高维融合和重复列而不稳定。D6 采用更窄的测试：

- 完全复用同一个 D4 head 和 D4 logits；
- 不增加特征列；
- 只根据位置、上一 action 或最近两次 action，使用不同 threshold；
- 小组数据不足时，把局部门槛向全局门槛收缩，减少过拟合。

这相当于问：是否只要根据“刚刚说过没有”轻微调整说话标准，就能稳定提高全局 Macro F1？

### 14.2 结果

| 版本 | Macro F1 | 相对 D4 |
|---|---:|---:|
| D4 global replay | `0.6846` | - |
| Position shrunk | `0.6855` | `+0.0009` |
| Last action shrunk | `0.6778` | `-0.0068` |
| Last two unshrunk | `0.6633` | `-0.0213` |
| **Last two shrunk，主版本** | **`0.6747`** | **`-0.0099`** |

主版本：

- bootstrap 区间 `[-0.01482,-0.00492]`，完全低于 0；
- 0/5 folds 为正；
- 0/4 domains 为正；
- 三组 stability split 分别 `-0.0080/-0.0019/-0.0066`；
- 修复 246 个 D4 错误，却新增 341 个，净减少 95 个正确决策。

### 14.3 为什么“每组都变好”，合起来却变差

这是 D6 最值得汇报的分析。

在 II、IS、SI、SS 每个小组内部单独计算 Macro F1 时，局部门槛看起来都提高。但官方 Macro F1 是把所有 chunk 的 TP/FP/TN/FN 合在一起后再计算，不能把各组分数直接平均。

一个简单例子：

- A 组把 20 个 silent 改成 interrupt，可能在 A 组多找回 8 个 TP；
- B 组做同样调整，也可能在 B 组看起来有益；
- 但两组一起会累计大量 FP，导致全局 silent precision 和 silent F1 明显下降。

D6 正是这样：增加了 47 个 TP，却同时增加 142 个 FP。局部目标与最终全局目标不一致，不是继续微调收缩常数就能可靠解决的问题。

### 14.4 路线结论

D6 是跨 fold、domain 和新 split 一致的负结果，因此 action-history 特征和门槛搜索在当前公共集上正式停止。D4 保持不变。

D6 同样只使用 CPU，正式运行约 444 秒，并完成独立第二次复现；核心比较、五组预测和三组稳定性结果逐字节一致。因此它是一个可复现的负结论，不是偶然运行失败。

## 15. 7 月 22 日：重新核对官方规则并重排路线

### 15.1 关于隐藏测试 `dialog` 的最新口径

- 官方公开 Proactive runner 每个 chunk 直接读取 `dialog[j]`；
- 公开 runner 不会把模型刚生成的答案写回下一 chunk；
- 公开 validation 中的 assistant 新增确实等于上一 gold action。

但当前公开的正式规则只说明隐藏测试由组织方运行部署包，没有明确写出隐藏 JSON 字段，也没有说明 assistant 历史是组织方预先提供，还是要求模型自行回写。官方 Docker 模板要到 8 月 8 日才发放。

因此需要修正汇报口径：

> “隐藏测试一定提供与公开集相同的标准 dialog”目前是高概率主假设，不是已经写入正式规则的保证。

这也是 D4 当前最大的外部风险。应尽快向组织方书面确认：

1. 隐藏输入是否包含与 intervals 等长的 `dialog[i]`；
2. 其中 assistant turns 是组织方提供的标准历史，还是模型自己的历史；
3. 推理入口是整段 session，还是逐 chunk 回调；
4. 去掉 `answers` 后还保留哪些字段。

### 15.2 Planner 的最终判断

`dialog` 和 planner 在概念上不是一回事：

- dialog 是“之前说过什么”的记录；
- planner 是“有哪些步骤、当前在哪一步、下一步是什么”的任务状态。

所以从长期研究看，dialog 不能完全代替 planner。但从当前竞赛证据和一周时间看，不应再训练学习型 planner：

- R1 小规模 oracle state 没有正收益；
- S0 Mean Macro `0.2891`，状态识别不足；
- U1-V/U2 说明主要问题是历史依赖和语言冷启动，不是已经定位的步骤状态缺失；
- D4 已经直接利用官方可见阶段信号；
- 排名只评价 interrupt/silent；
- 新 planner 会带来参数、延迟、训练和误差累积风险。

建议最终只保留零参数的 planner-lite 状态：当前 chunk、累计 assistant turns、上次发言位置、最近自己的决策和输入模式，不生成自由文本计划。

### 15.3 R2-R5 重新定位

| 原阶段 | 当前决定 | 原因 |
|---|---|---|
| R2 粒度 | 移到赛后 | 还没有证明模型能可靠利用 state |
| R3 predicted state | 继续冻结 | S0 失败，S1 标注远未完成 |
| R4 noisy plan | 本轮取消 | 没有可部署 planner 可做噪声测试 |
| R5 训练选择 | 现在收口 | 选择 D4 轻量监督决策头，不做 GRPO |

### 16.3 这不是“视觉完全没用”

遮蔽视频会明显改写文本，删除当前画面也会改变具体名词和步骤，因此视觉确实参与计算。正确结论是：

> 视觉目前是一个不稳定的内容修正来源，还没有成为可靠的步骤状态解码器或正文启动条件。

不能把“视觉没有稳定提升 coverage”夸大成“模型完全不看图”；也不能把“文字变了”夸大成“视觉 grounding 已经正确”。

## 17. 当前遇到的主要问题和困境

### 17.1 排行榜目标与真实助手质量不完全一致

官方 C1 Macro F1 只检查 `$interrupt$/$silent$`。一句错误指导和一句正确指导，只要都以 `$interrupt$` 开头，在当前排名指标中没有区别。

因此 D4 可以是优秀的竞赛决策器，但正文质量仍只有中等甚至较低。汇报时应把“决策成绩”和“真实帮助质量”分开。

### 17.2 合法因果信号不等于真实部署稳健

上一轮 dialog 在时间上可见，所以 D4 没有未来泄漏。但公开 dialog 带有上一 gold action 痕迹；真实系统若使用自己的上一轮预测，错误会反馈到后续历史。

这叫“竞赛输入内合法，但部署分布可能不同”。D4 可以用于比赛，但不能把它描述成已经解决真实长期状态跟踪。

### 17.3 公共验证集被反复用于开发

D3、D4、D5、D6 都采用 session OOF 和冻结门槛降低过拟合，但仍然使用同一公开 validation。尤其 D4 来源于机制诊断，不能当成独立隐藏集证据。

这也是为什么 D5 即使高出 `0.0066`，仍按预设规则拒绝，而不是为了追最高数字继续调。

### 17.4 人工评分本身也有测量误差

Groundedness 的双人 kappa 只有 `0.0508`。若评分者对“当前画面支持”的定义不同，再多做几百条人评也可能只是放大尺度差异。

正确做法是先用明确事实逐条校准 rubric，再继续人评；在最终一周里，这项工作的优先级低于提交闭环。

### 17.5 Planner 缺少可靠输入和监督

公开数据没有 current step、completion cue 或 recovery plan 字段；S0 又表明 zero-shot 解码失败。继续做 planner 需要大量新标注和独立评估，不可能在一周内同时保证质量、稳定性和提交安全。

### 17.6 工程与合规仍有外部依赖

- 隐藏 `dialog` 契约未正式写入规则；
- 官方 Docker 模板 8 月 8 日才发；
- 项目顶层许可证未选择；
- starter kit 是 CC-BY-NC-4.0，如何放入获奖所需开源包要确认；
- 同学的 frame/history 输入策略尚待独立验证；
- 当前没有任何外部 leaderboard 或 registry 上传。

## 22. 术语的通俗解释

- **OOF**：把数据分组，每条样本只让“没有用过它训练的模型”预测，减少在训练集上自我评分的问题。
- **Bootstrap 区间**：反复换一批 session 重新算增益，看提高是否依赖少数视频。区间跨 0 表示不能排除换一批数据后没有收益。
- **Full-fit**：用全部公共数据训练一个最终模型。它适合部署，但在同一数据上的分数会偏乐观。
- **Fallback**：模型没有生成具体正文时使用的固定句，目前是 `Please continue with the next step.`。
- **Dialog-stage**：不理解句子含义，只统计已经说过几次、多久没说、上一轮是否新增发言等对话阶段信息。
- **Causal**：当前决策只使用当前和过去能看到的内容，不看未来视频、未来 dialog 或当前答案。
- **Self-fed**：下一时刻历史由模型自己的上一轮输出组成，而不是组织方预先给的标准历史。
- **Planner-lite**：只维护 chunk、最近发言和动作等确定性状态，不运行另一个模型生成任务计划。

## 23. 最终总结

7 月 17 日至今最有价值的成果不只是把公共 OOF 从 `0.6341` 提到 `0.6846`，而是逐层弄清了提升来自哪里、哪些方向不值得继续：

- D3 证明跨时间信息有用；
- D3-D 证明主要可利用信息来自官方 dialog 阶段；
- D4 把这一发现做成了单一、可运行、满足 Small 参数限制的提交候选；
- U1、S0、U1-V、U0 和 U2 共同证明当前 planner/state/视觉 grounding 还不足以成为提交主模块；
- D5 证明短期动作历史有一点额外信号，但不够稳定；
- D6 证明按历史分组单独调门槛不是正确的全局优化方式；
- 负面实验严格按预设门槛停止，避免为了追最高公共分继续过拟合；
- 工程上已经完成 D4 adapter、参数核算、全量在线回放、GPU 等价 smoke 和回归测试；
- 下一步应把时间投入隐藏输入确认、无 dialog 退路、许可证、容器和外部提交，而不是再开启一个新 planner。

这使当前项目从“不断试模型”进入了“有证据地选择模型并完成交付”的阶段。
